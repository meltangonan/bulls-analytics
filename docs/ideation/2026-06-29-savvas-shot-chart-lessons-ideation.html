What is already true

Grounding Context

Codebase Context

The repo already fetches NBA shot rows, draws the half-court, supports raw makes/misses and FG% hexbins, computes detailed 12-zone summaries and PPS, and produces branded feed graphics with cached headshots.

Existing product direction

CONTENT_IDEAS.md already names tonight-vs-season, efficiency gap, shot diet, postgame charts, and league context. The ideas below favor moves that make several of those cheaper or sharper.

What the article contributes

Savvas makes the coordinate system, orientation check, court geometry, scatter, KDE, hexbin, and presentation layers tangible. The durable lesson is to separate data, geometry, and visual encoding.

What has changed since 2015

The repo correctly uses nba_api instead of hand-building endpoint URLs. Current endpoint docs also expose a league-average table and contextual shot fields that the normalized repo tables currently discard.

Feedback in one sentence: do not add Seaborn or a KDE chart just because the tutorial used them; the strongest gap is a clean distinction between where shots happen, how valuable they are, what baseline they differ from, and how much evidence supports the claim.

1. League-relative efficiency-gap court

Analytical meaningConfidence 95%Medium complexity

Description: Show points gained or lost versus league-average shooting from the same court areas. Start with the repo’s explicit zones, not a spatial model: attempt volume determines the weight, league-relative conversion determines the gain or loss, and the output highlights the handful of zones that materially move the total.

Basis
direct: CONTENT_IDEAS.md already calls this “The Efficiency Gap”; the official endpoint returns a league-average dataset, while get_team_shots() currently selects only the first returned frame.
Rationale
This turns a descriptive picture into the visceral question “where are the Bulls gaining or leaving points on the table?” It is analytically meaningful, naturally supports one or two callouts, and can use a cheaper league baseline for basic zone work.
Downsides
The endpoint baseline must be mapped carefully to the repo’s 12-zone vocabulary. Custom PPS rankings may still require the existing 30-team pull.
Why it survived
It is already desired, grounded in available data, and more decision-changing than another raw heatmap.
2

2. “Tonight vs normal” residual profile

Analytical meaningConfidence 93%Medium complexity

Description: Compare the latest game’s shot profile with the season baseline and show only the largest changes in shot share and PPS. The result answers “what was different tonight?” instead of presenting two dense charts and asking readers to spot the difference themselves.

Basis
direct: The repo already supports get_team_shots(last_n_games=1) and season shots, and the existing content brief explicitly proposes “Tonight vs Season.”
Rationale
A generic profile comparison can also support first-20 vs last-20, Bulls vs opponent, and current roster vs all players. One analysis primitive can therefore drive several recurring notebook ideas.
Downsides
Single-game zone rates are noisy. The chart must show attempts and avoid turning tiny samples into strong color or copy.
Why it survived
It has a clear recurring job, uses existing fetches, and produces a stronger editorial answer than a raw postgame scatter.

3. Shot-profile core plus explicit visualization modes

Notebook explorationConfidence 91%Medium complexity

Description: Create one tidy, testable shot-profile table—one row per detailed zone with FGA, FGA share, FGM, FG%, PPS, and games—then make charts thin consumers with named modes such as scatter, frequency, efficiency, and later relative_value. Keep raw makes/misses for single games and diagnostics.

Basis
direct: Zone aggregation is currently spread across detailed_zones(), points_per_shot(), game_zone_stats(), league helpers, and the plotting function. shot_chart(show_zones=True) silently means mean FG% hexbin.
Rationale
The profile table is the reusable analysis seam; explicit chart modes make the notebook question visible in code. Together they prevent the visualization module from growing into one-off builders.
Downsides
This touches a public helper. Preserve show_zones temporarily or update notebooks and docs in one small migration.
Why it survived
It compounds across nearly every strong content direction while fitting the repo’s analysis-first convention.

4. Trustworthy bivariate shot map

Correctness and testsConfidence 86%Medium complexity

Description: Use fixed bins, marker area for attempts, and diverging color for PPS or FG% versus a baseline. Mask or fade low-attempt bins, validate left/right orientation with known landmarks, and test the underlying bin table plus the Matplotlib collections—not only that a figure exists.

Basis
external: Dan Vatterott’s follow-up to Savvas combines size for shot count and color for shooting percentage. direct: the current heatmap encodes only mean FG% with mincnt=1, and its tests assert only that a Figure is returned.
Rationale
The map can answer volume and performance together, but only if sample size and orientation are treated as first-class semantics. That trust layer is what makes the richer encoding worthwhile.
Downsides
Bivariate legends are harder to read, especially on mobile. Prototype in a notebook and reject it if a paired two-panel view is clearer.
Why it survived
It is the most faithful extension of the article and its follow-up, with the statistical caveats the originals largely omit.
5

5. Roster-transition identity map

Analytical meaningConfidence 85%Low complexity

Description: Use the repo’s mandated all-players and current-roster views as an explicit comparison: which zones, volumes, and values disappeared or shifted after the trades? This turns a filtering caveat into a Bulls-specific story about how the team’s offensive identity changed.

Basis
direct: AGENTS.md requires both all-player and current-roster views because the API includes shots from traded players.
Rationale
The comparison is unique to the repo’s live context and can be produced first as a concise notebook. It also stress-tests the general profile-comparison idea with a meaningful real case.
Downsides
A roster filter alone mixes personnel and time effects. The notebook should label the comparison honestly or use a date split when answering before-versus-after.
Why it survived
It transforms an awkward data detail into a concrete, timely analytical narrative.
6

6. Richer, consistent shot-row contract

Data contract and contextConfidence 83%Low complexity

Description: Make player and team shot tables agree on core zone fields, then retain a small set of optional context columns—action_type, period, time remaining, opponent/home-away identifiers, and event ID—when the endpoint provides them. Store the endpoint’s league-average frame through a separate, clearly named path rather than changing every existing return value at once.

Basis
direct: team shots add shot_zone_area and player identity while player shots do not. external: official endpoint docs list ACTION_TYPE, period/time, matchup identifiers, and the league-average dataset.
Rationale
Schema parity lets the same zone helpers work for player and team notebooks. Retaining cheap contextual fields preserves future analytical options without committing to clutch or play-type features today.
Downsides
Every retained field becomes a contract to test. Keep the list narrow and tolerate absent upstream columns.
Why it survived
It is a small leverage move grounded in data already arriving from the API.
7

7. Comparable shot fingerprints

Social storytellingConfidence 78%Medium complexity

Description: Render the same compact 12-zone silhouette for each player, roster scope, or period: attempt share on one layer and PPS or relative value on another. Use small multiples with identical scales so the viewer can recognize and compare offensive “fingerprints.”

Basis
reasoned: fingerprints work because fixed measurement positions make profiles recognizable; this repo already owns a stable 12-zone vocabulary and needs both all-player/current-roster views.
Rationale
This creates a reusable comparison format for player fit, roster change, and first-half/second-half identity without inventing a new metric.
Downsides
Small multiples can become cramped on a phone. The first prototype should compare only two or three profiles.
Why it survived
It covers the social-communication axis with a Bulls-specific, comparison-first format rather than decoration.
