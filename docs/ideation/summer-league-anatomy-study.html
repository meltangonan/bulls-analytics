Anatomy of an F5-quality sports graphic

A structural study of three Owen Phillips tutorials, measured against the current Summer League report. The goal is to borrow the method—data shape, hierarchy, rhythm, and finishing discipline—without copying The F5's visual identity.

Study only · no current Summer League work overwritten

Executive finding

The quality gap is not “R/Python versus design software.” Owen chooses a layout grammar first, reshapes the data to serve it, and gives only one element the strongest emphasis. Our current report starts with a complete set of components, then asks spacing and styling to manufacture hierarchy afterward.

Reference anatomy 1

The repeated-row board

TITLE + one-sentence metric definition
SOURCE

What the code is doing

1A reusable theme sets the background, font, title, subtitle, caption, and default spacing.
2Each player becomes a facet, so the plotting system enforces identical row geometry.
3One rectangle creates the right-side value zone for every row.
4Name and team are stacked as one identity unit; headshot and logo do labeling work.
5A single-hue gradient encodes the ranked stat. No other element competes with it.
6The image is exported directly at the intended social aspect ratio and high DPI.

Transferable lesson: repetition creates polish before decorative styling begins.

What the code is doing

1Fetch and clean only the columns needed for the table.
2Combine makes and attempts into compact strings so the table can breathe.
3Sort by DRE and make the ranking logic explicit.
4Use a table engine for column widths, baselines, row rhythm, formatting, and cropping.

Transferable lesson: dense information can still have one focal column.

Reference anatomy 3

The players-as-columns comparison

LEFT-ALIGNED TITLE + coverage window

What the code is doing

1Unpivot and pivot the data so the comparison question determines the table orientation.
2Use image headers so faces and names become one column-label unit.
3Center player values, left-align stat labels, and let zebra rows handle tracking.
4Avoid extra stat cards, icons, or color when the comparison itself is already clear.

Transferable lesson: reshape the data before styling the output.

Reference anatomy 4

The shot chart as a coordinate system

TITLE + exact coverage window
ONE FGA MARKER · SOLID MAKE / HOLLOW MISS

What the tutorial is doing

1Load reusable NBA court paths instead of drawing a decorative court by eye.
2Convert raw locX/locY into the same units and origin as the court geometry.
3Filter play-by-play events to field-goal attempts. The shooting-foul layer is separate and is excluded from our study.
4Use a fixed coordinate ratio and explicit court limits so spatial patterns are not distorted.
5Overlay point marks, remove axes and grid scaffolding, and let the court provide orientation.
6Use low alpha for season-level density; use stronger make/miss marks for one-game charts.

Our adaptation: every FGA uses the same circular marker. Solid means make and hollow means miss. The simpler encoding is easier to read and can later map cleanly to Bulls red.

Current-state audit

Why the Summer League carousel feels designed in pieces

The current prototype is ambitious and analytically sound, but it contains several nested visual systems. The script is 804 lines and uses 47 separate text placements. That produces control, but it also makes hierarchy depend on dozens of unrelated coordinate decisions.

Wireframe study

Three structures using the same settled data

A · Repeated-row board

Fastest scan and strongest cover. Each player has the same row grammar; points own the endpoint.

B · Comparison table

Cleanest direct comparison. Best as a self-contained report or reference slide, weaker as a narrative carousel.

C · Editorial player story

Strongest analytical slide. One claim owns the page; the shot chart proves it and three figures support it.

Study verdict: Direction A is the best cover grammar. Direction C is the best player-slide grammar. Direction B is useful, but it solves a different job: compact comparison rather than editorial storytelling.

Technical finding: Great Tables 0.22 and gt-extras 0.0.8 work in the project virtual environment. Current gtsave() rendered both PNGs successfully. The packages remain uncommitted environment-only dependencies until a selected table format justifies adding them to requirements.txt.

Caution: gt-extras documents itself as still in development and warns that breaking changes should be expected. Use it in a prototype first; keep a fallback path for durable formats.

Recommendation

Build the carousel as A + C2

Slide 1: use the repeated-row board. Keep the score/date as context, then show the three players with one short takeaway and one emphasized endpoint. Remove the team dashboard and team shot map from the cover.

Slides 2–4: use one editorial player story per slide. Choose one thesis from the existing data, let the exact shot chart prove the spatial claim, and limit the support rail to a few numbers.

Direction D: use the shared-scale small multiple only when comparing shot profiles is itself the point. It should replace separate player slides, not be added on top of them.

Direction B: preserve the Great Tables comparison as a separate format option. It may become the right answer for a “three-player report card” or a one-slide post, but it should not be forced into this narrative carousel.
